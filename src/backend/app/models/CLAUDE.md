# models — Pydantic schemas & enums

Every request/response body and every on-disk JSON document corresponds to a Pydantic model. The same models validate the API boundary **and** validate documents at book load (load-time schema validation; violations treated as corruption per doc 03). Treat this as the OpenAPI components section.

Parent: [app](../CLAUDE.md). Specs: [04 API §2](../../../../docs/claude-tech-specs/04-api-reference.md) (shared objects + enums), [03 Data](../../../../docs/claude-tech-specs/03-data-storage.md) (on-disk schemas).

## Enums (doc 04 §2.1)

`provider` (anthropic · openai · gemini · openai-compatible · ollama) · `outputType` (chat · edit-proposals · metadata-proposals) · `placement` (trunk · unanchored · floating · orphan · archived — **computed**, never stored) · `sceneStatus` (active · archived) · `relationshipType` (before · after · around) · `todoStatus` (open · done · closed) · `todoOrigin` (user · dependency · ai) · `conversationKind` (note · chat · ai-job · bookkeeping · task-discussion) · `conversationStatus` (open · queued · running · waiting · done · failed · archived — the run lifecycle; there is no separate Job) · `enrichScope` (summary · characters · both — resolved at creation into one bookkeeping conversation per field) · `proposalType` (edit · metadata-update · todo-create · character-create · character-relationship-create · resource-create) · `proposalStatus` (pending · applied · rejected · not-found) · `parentType` (scene · chapter · part · book) · `characterRelationshipCategory` (family · romantic · friendship · rivalry · professional · mentorship · other) · `gitFileStatus` (modified · added · deleted · untracked · renamed).

## Shared objects (doc 04 §2.2)

`ModelConfig` (responses carry `apiKeyMasked`, never the real key) · `ModelTestResult` (live model check: `ok`, `message?`, `error?`, `latencyMs?`) · `AIJobDefinition` · `Placeholder` · `BookSummary` · `Book` · `Part` · `Chapter` · `Scene` (assembled API response — full flat shape; `seq`/`placement` computed on read; `chapterId` XOR `partId`) · `SceneRecord` (persisted master row, trimmed to identity/hard-chain/structure — `db/scenes.json`) · `SceneMeta` (persisted `scenes/{id}/meta.json` — soft narrative metadata) · `SceneBookkeeping` (persisted `scenes/{id}/bookkeeping.json` — AI-owned + content-derived) · `SoftRelationship` (persisted per-scene, `scenes/{id}/relationships.json`) · `Dependency` (persisted per-scene, `scenes/{id}/dependencies.json`; model exists but no CRUD API yet) · `Character` (identity + craft fields; + computed `sceneCount`) · `CharacterRelationship` (directional `aToB`/`bToA` between two characters) · `Plotline` (+ `sceneCount`) · `Todo` (+ resolved `parentTitle`) · `ResourceFile` (response-only — `{ filename, mimeType, sizeBytes, updatedAt }`; never persisted, since `resources/` is scanned, not indexed) · `ConversationSummary` (index row; carries `status`) · `Conversation` (the run entity) · `Message` · `Proposal` (payload variants: edit / metadata-update / todo-create / character-create / character-relationship-create / resource-create) · `AiJobRunRequest`/`AiJobRunResponse` (`{ conversationId }`) · `EnrichRequest`/`EnrichResponse` (`{ conversationIds }`) · `GitFile` · `GitStatus` · `CommitInfo` · `CheckItem` · `CompileReport`.

## ID scheme (doc 03)

`{prefix}-{6 lowercase hex}` via `secrets.token_hex(3)`, collision-checked per collection. Prefixes: `bok scn chp prt chr plt cnv msg tdo dep rel prp mdl aij` (`aij` = AI-Job *definition* in `app.json`; there is no run-record prefix — the conversation `cnv` is the run). Sentinels `scn-START` / `scn-END` are reserved, recordless, and valid relationship endpoints.
