# services — business logic

All logic lives here. Routers delegate; services enforce business rules, own the per-book asyncio mutation lock, and drive persistence via BookDataManager. Reads take no lock; mutations replace collections rather than tweaking in place so readers never see half-mutated state.

Parent: [app](../CLAUDE.md). Spec: [04 API §1.3](../../../../docs/claude-tech-specs/04-api-reference.md) (service table), plus docs 03/05.

## Service catalog

| Service | Responsibility |
|---|---|
| `SettingsService` | `app.json` read/write; model-config + AI-Job validation |
| `PlaceholderRegistry` | Defines placeholders; validates prompts; resolves `@tokens` at run time against an explicit `scene_id` (doc 05) |
| `ContextAssembler` | Builds LangChain message lists; injects CURRENT SCENE when callers pass parent scene; resolves `@tokens` in user messages for the model only (doc 05) |
| `BookScanner` | Scans `booksHome`; caches shelf; reads each `config/book.json` |
| `BookService` | Book creation (scaffold, git init, initial commit), rename/cover, folder rename |
| `BookDataManager` | One per open book. Loads `db/*.json` + `config/book.json` into memory; owns the book's asyncio mutation lock; atomic write-through; conversation files + derived index; `ui.json`. Also owns the per-scene folder (`scenes/{id}/meta.json` / `bookkeeping.json` / `dependencies.json` / `relationships.json` / `todos.json`) — lazy-loaded and cached per scene, with a live in-memory reverse index for dependencies (**not** for todos — `get_all_scene_todos()` reads fresh each call, doc 03 §Todos storage split), and transparent one-time migration of old-shape (pre-split) `db/scenes.json`. `find_todo(id)` resolves a bare todo id to whichever storage tier it lives in; `list_conversations_by_status(status)` filters the derived index in memory (how the conversation worker finds `queued` runs). One-time cleanup deletes the legacy `db/jobs.json` on load (the conversation is the run now — there was never anything to migrate). Every `save_*` fires `book-changed` on the EventHub — the single chokepoint every mutating service already funnels through, so no service needs its own post-write hook |
| `ChainService` | Hard-chain algebra: splice, heal, walk, seq/placement computation, contiguity + completeness checks. Reads only `id`/`status`/`previousSceneId`/`nextSceneId` off scenes — unaffected by the master/per-scene split |
| `SceneService` | Scene CRUD; routes each field of a `SceneUpdate` to the master table, `meta.json`, or `bookkeeping.json` depending on which it belongs to, and assembles the flat `Scene` API response back out of the three (doc 03); content saves (hash, word count — bookkeeping.json only, never touches the master table); dependency-todo fanout (creates scene-parented `Todo`s, deduped by open `sourceDependencyId`); enrichment settle timers; file naming/renames |
| `TodoService` | CRUD for todos, routed by `parentType` to whichever storage tier owns it — book-level `db/todos.json` for chapter/part/book, or the owning scene's `scenes/{id}/todos.json` for scene (doc 03 §Todos storage split); resolves `parentTitle` for the API response |
| `StructureService` | Parts/chapters linked lists; move-before/after rewiring; blocked deletions; plotlines; characters (uniqueness) |
| `ConversationService` | The single run entity — notes, chats, AI-Job runs, and bookkeeping runs are all conversations. Owns their status lifecycle (`open`→`queued`→`running`→`waiting`/`done`/`failed`); `send_message` streams via AIOrchestrator and, for run kinds, records the outcome (a bookkeeping reply with no tool call → `waiting`, i.e. the AI asked the author something). `body=None` generates against the thread as-is (how the worker triggers a queued run). Utility-model title when Untitled; hard delete; emits `conversation` SSE on every transition |
| `AiJobService` | Turns a saved `AIJobDefinition` into a ready conversation: resolves the prompt's `@placeholders`, opens an `ai-job` conversation with it inside, at `open`. No model call, no run record — the author reviews and sends; `ConversationService` runs it |
| `ProposalService` | Locate/apply/reject proposals — a code path that mutates on behalf of AI output after author acceptance; accepted `todo-create` proposals go through `TodoService.create`, with `conversationId` set to the proposing conversation; accepted `resource-create` proposals go through `ResourceService.create_text_file` |
| `ResourceService` | Files beside the book (`resources/`) — research, references, worldbuilding notes. No id, no index: scans the folder fresh on read, same reasoning as `BookScanner` (doc 01 hard rule 4, portability). Uploads and accepted `resource-create` proposals both suffix on a filename collision rather than overwrite; deletion moves to `.trash/`, never unlinks. Module-level helpers (`resources_dir`, `safe_resource_path`, `is_text_resource`, `scan_resources`) are reused directly by the `list_resources`/`get_resource` read tools, so the tool registry needs no new dependency |
| `EnrichmentService` | A **creator only**: builds the summary/character prompts and opens `queued` bookkeeping conversations (`both` → two, one per field, each on its own configured model). It runs no model and writes no fields — the conversation does, via the execute tools. No settle timer; never scheduled from autosave |
| `AIOrchestrator` / `ModelFactory` | Model invoke_once / structured / stream + tool loop |
| `ToolRegistry` | Read tools (execute freely, read-only) + propose tools (accumulate proposals) + **execute tools** (write bookkeeping directly, bound to one scene, opt-in for bookkeeping conversations only) |
| `GitService` | GitPython wrapper: status/stage/unstage/discard/diff/commit/push/pull/log. `status()` builds the badge's `summary` string (`"all-changes-synced"` / `"7-new, 1-updated, 3-deleted"`) and emits nothing — callers decide when to broadcast. Mutating ops emit `git-status` immediately in-request; `discard` also `BookRegistry.forget`s the book so discarded disk state is reloaded. Incidental dirtying is picked up by the [git-status worker](../worker/CLAUDE.md) instead. Never run inline on a write path |
| `CompileService` | Completeness check (errors/warnings); build to `compiled-book/` |
| `EventHub` | Per-book SSE pub/sub (lives in [core](../core/CLAUDE.md)); all services emit through it |

## Key invariants

- **Prose hard rule:** only `SceneService` content path and `ProposalService` (edit apply, author-triggered) write scene **`.md`** files. AI cannot touch prose. It *can* write scene **bookkeeping** (summary, character involvement) — via the execute tools in a bookkeeping conversation, under the standing consent the Bookkeeping toggles grant — because bookkeeping is not prose. Propose tools still only emit proposals; execute tools go through `SceneService.update_scene` (lock + validation), never the manager directly. Resource files fall into neither carve-out, so they get **no** execute tool at all — `propose_resource_create` only accumulates a proposal, and `ProposalService._apply_resource_create` is the sole write path, gated on author Accept.
- **Single writer / lock:** every mutation runs under the target book's asyncio lock (`app.json` has its own).
- **Atomic persistence:** serialize → `{file}.tmp` → flush + fsync → `os.replace`; POSIX also fsyncs the directory. Never overwrite a file that failed to parse at load (quarantine to `{file}.corrupt-{ts}`).
